# 领星自动化当前规则说明

更新时间：2026-04-27

本文档用于固化当前已经实现并验证过的“领星自动化下载 + Excel 整理 + 代码生成输出”规则。后续如果业务规则再调整，应优先更新本文档与处理脚本，保持一致。

## 1. 适用文件

当前流程基于以下文件：

- `rpa.docx`
- 一个或多个领星下载的仓库明细 Excel
- `MSKU对应品线表.xlsx`
- `店铺明细表.xlsx`
- 一体化入口脚本：`lingxing_rpa_runner.py`
- 处理脚本：`lingxing_excel_processor.py`

## 2. RPA 下载流程规则

下载流程以 `rpa.docx` 为准，当前确认的主流程如下：

1. 打开领星并登录：`https://erp.lingxing.com/erp/home`
2. 进入 `FBA`
3. 点击 `FBA货件`
4. 按货件号搜索对应货件
5. 进入对应仓库明细
6. 点击下载装箱清单
7. 选择“导出不包含图片”
8. 将下载的 Excel 作为后续整理输入

说明：

- 如果 `rpa.docx` 与 Excel 处理规则冲突，优先记录冲突并人工确认，不允许凭空猜测。
- 当前已支持“网页提交任务 + Docker worker + headless Playwright 下载 + Excel 整理”一体化执行；其中下载步骤由 `lingxing_rpa_runner.py` 驱动，Excel 处理仍由 `lingxing_excel_processor.py` 负责。

### 2.1 一体化入口规则

一体化入口脚本固定为：

- `lingxing_rpa_runner.py`

脚本职责如下：

1. 读取 TXT 或 Excel 格式的 FBA 清单
2. 启动 Playwright 浏览器，优先使用持久化登录态
3. 复用已存在的领星登录态；若失效则优先使用本地配置自动登录
4. 逐个 FBA 搜索并进入详情页
5. 在 `箱子标签` 页面逐仓下载“导出不包含图片”的装箱清单
6. 每个 FBA 下载完成后，立即调用 `lingxing_excel_processor.py` 的现有整理逻辑
7. 在批次目录输出每个 FBA 的自动化报告，以及整个批次的汇总报告

### 2.2 登录配置规则

本地登录配置文件默认命名为：

- `lingxing_rpa.local.json`

文件结构固定为：

```json
{
  "username": "你的领星账号",
  "password": "你的领星密码"
}
```

登录顺序固定为：

1. 先复用自动化专用 Playwright profile 中已有的登录态
2. 如果仍停留在登录页，则尝试读取 `lingxing_rpa.local.json` 自动填充账号密码
3. 如果登录后仍需要验证码、短信校验或额外人工操作，则暂停等待人工处理，完成后继续

约束：

- 不允许把账号密码硬编码进 Python 脚本
- 自动化使用专用持久化 `user-data-dir`，不占用用户日常 Chrome 配置

### 2.3 清单输入规则

一体化入口的 `manifest` 当前支持：

- `.txt`
- `.xlsx`

TXT 规则：

- 一行一个 FBA 号
- 忽略空行
- 忽略以 `#` 开头的注释行
- 去重但保持原始顺序

Excel 规则：

- 仅读取第一个工作表
- 优先识别表头：`FBA号` / `货件单号` / `FBA`
- 如果未识别到上述表头，则回退读取第一列
- 去重但保持原始顺序

## 3. 源文件选择与票序规则

### 3.1 源文件识别

在当前 FBA 的来源目录中读取所有符合以下条件的 Excel：

- 文件扩展名为 `.xlsx`
- 文件名以 `FBA` 开头
- 排除临时文件，例如 `~$` 开头的文件

优先选择文件名中包含 `_NO_PIC` 的文件；如果没有，再退回到所有 `FBA*.xlsx` 文件。

系统会先按 Excel 内容识别源文件结构，但源结构只用于决定“怎么解析”，不再用于决定最终输出类型：

- `ONE_SKU`：表头包含 `发货量`、`单箱数量`、`箱数`、`箱号`，表示每箱单款 SKU。
- `MUL_SKU`：表头包含 `发货数量`、`已装箱数`，并存在 `第1箱`、`第2箱` 这类箱列，表示每箱多款 SKU。

最终输出分流以源文件内部 `货件名称` 为准：

- `货件名称` 包含 `UPS`，大小写不敏感：走第二种 `装箱信息` 输出。
- `货件名称` 不包含 `UPS`：走第一种 `货代信息表` 输出。
- 同一个 FBA 下载目录内的多个 UPS 文件合并生成一份装箱信息工作簿。
- 不同 FBA 之间绝不合并，因为每个 FBA 都在自己的下载目录内独立调用处理函数。
- 如果同一个 FBA 下载目录里同时存在 UPS 和非 UPS 文件，会分别输出装箱信息表和货代信息表。

### 3.2 票序规则

票序按来源文件的创建时间排序；若时间相同，再按文件名排序。

排序后的第几个文件，就是第几票：

- 第 1 个文件：第 1 票
- 第 2 个文件：第 2 票
- 第 3 个文件：第 3 票
- 以此类推

## 4. 输出结构与代码生成规则

### 4.1 无模板生成规则

当前输出工作簿不再依赖 `模板.xlsx`。  
`lingxing_excel_processor.py` 会直接在代码中生成目标工作簿、表头、票块布局和样式，输出字段固定为：

- `NO.`
- `SKU`
- `工厂型号`
- `品名`
- `内盒标签`
- `箱号`
- `套`
- `箱数`
- `套/箱`
- `票数`
- `FBA号`
- `备注/品线`
- `说明书/包装上-品牌`

代码生成的默认结构固定为：

- 工作表：按总票数命名，例如 `1票`、`3票`
- 标题区：`A:M` 合并
- 表头区：第 `2-3` 行
- 数据起始行：第 `4` 行

`货件名称` 包含 `UPS` 的文件会生成装箱信息表，不依赖外部模板文件，文件名固定为：

`装箱信息-{货件名称}-{货件单号}.xlsx`

例如：

`装箱信息-UPS-HIA1-9809-5CTN-15PCS-260423-FBA19C2P8D5D.xlsx`

装箱信息表核心结构固定为：

- 标题：`装箱单（店铺简称）第X票-FBA号`
- 工作表名：`共N票{总发货量}件`
- 顶部信息：美西时间、仓库地址、发往国家
- 明细列：`NO.`、`MSKU型号`、`工厂型号`、`箱号`、`套`、`箱数`、`套/箱`、`条码`、`箱号`
- 连续箱内 SKU 组合完全一致时，会合并为 `编号01-03` 这类箱组
- 混装箱内多 SKU 时，第一行写箱号、箱数、套/箱、外箱号，其余 SKU 行跟随在下方
- 仓库地址使用黄色底、红字、字号 12
- 条码列使用黄色底、红字
- 含中文的单元格字体使用宋体，不含中文的英文/数字单元格字体使用 Arial

### 4.2 输出工作簿结构

输出时只保留 1 个工作表。

工作表名称按总票数命名：

- 1 个源文件：`1票`
- 3 个源文件：`3票`

### 4.3 多票排版规则

多票数据不拆分成多个工作表，而是在同一工作表内纵向堆叠。

每一票使用一个独立票块：

1. 第 1 行：标题
2. 第 2-3 行：表头
3. 第 4 行开始：明细
4. 明细结束后追加 1 行汇总
5. 下一票紧接上一票汇总行之后开始

以 `N` 行明细为例，某票块的结构为：

- 标题行：`start_row`
- 表头行：`start_row + 1`
- 表头下方空白合并行：`start_row + 2`
- 明细起始行：`start_row + 3`
- 汇总行：`start_row + 3 + N`
- 下一票起始行：`start_row + 4 + N`

## 5. 字段映射规则

| 输出字段 | 来源字段/来源规则 | 当前实现规则 |
| --- | --- | --- |
| `NO.` | `序号` | 直接填写 |
| `SKU` | `MSKU` | 直接填写 |
| `工厂型号` | `SKU` | 直接填写 |
| `品名` | `品名` | 直接填写 |
| `内盒标签` | `FNSKU` | 直接填写 |
| `箱号` | `箱号` | 先清洗，再写成 `编号 x` 或 `编号 x-y` |
| `套` | `发货量` | 直接填写 |
| `箱数` | `箱数` | 直接填写 |
| `套/箱` | `单箱数量` | 直接填写 |
| `票数` | 查表后生成 | 填 `店铺简称海运第x票`，如 `WK海运第2票` |
| `FBA号` | `货件单号` | 按规则提取 |
| `备注/品线` | `品名` | 按关键词归类为 `浴缸 / 厨房 / 淋浴 / 面盆` |
| `说明书/包装上-品牌` | 查表后生成 | 使用店铺名称并做英文首字母大写格式化 |

重要说明：

- `备注/品线` 不是取 `MSKU对应品线表.xlsx` 中的原始品线，而是按 `品名` 做四类归类。
- 文件名中的“品线”则使用 `MSKU对应品线表.xlsx` 中查到的原始 `品线` 值。

## 6. 箱号清洗规则

### 6.1 目标格式

输出时统一写为：

- 单个箱号：`编号 3`
- 范围箱号：`编号 1-2`

### 6.2 识别规则

1. 去掉前缀、字母和无意义前导零
2. 识别末尾数字
3. 范围格式统一转为 `起始数字-结束数字`

支持的常见形式：

- `FBA19B9KWNFHU000001～2` -> `编号 1-2`
- `FBA19B9KWNFHU000003` -> `编号 3`
- `FBA19B9KWNFHU000001-000002` -> `编号 1-2`
- `FBA19B9KWNFHU000001~000002` -> `编号 1-2`

如果无法识别：

- 保留原值
- 记录异常

## 7. FBA号处理规则

`FBA号` 来源于下载文件中的 `货件单号`。

处理规则：

1. 如果整个值本身就是清晰的字母/数字/连字符组合，直接使用
2. 如果拆分后只得到 1 个明显编号，也直接使用
3. 如果存在多段且无法确定哪一段是目标编号，则保留原始值并记录异常

当前常见结果如：

- `FBA19B9KWNFH`
- `FBA19B9HVNWG`
- `FBA19B9LLTB1`

## 8. 店铺与品牌查表规则

### 8.1 查表链路

对每一行数据按以下顺序查表：

1. 取当前行 `MSKU`
2. 到 `MSKU对应品线表.xlsx` 里按 `MSKU` 精确匹配
3. 从匹配结果读取：
   - `店铺`
   - `品线`
4. 将 `店铺` 视为 `店铺+站点` 主键
5. 到 `店铺明细表.xlsx` 里按 `店铺+站点` 精确匹配
6. 从店铺明细表读取：
   - `店铺`
   - `店铺简称`
   - `店铺+站点`

### 8.2 `说明书/包装上-品牌` 规则

`说明书/包装上-品牌` 使用店铺明细表中的 `店铺` 字段，并进行英文格式化：

- `WINKEAR` -> `Winkear`
- `WALKER DIRECT` -> `Walker Direct`

中文、数字、符号保持原样。

### 8.3 `票数` 规则

`票数` 由店铺简称和票序生成：

- 规则：`店铺简称海运第x票`
- 示例：`WK海运第1票`

如果同一票内存在多个店铺简称：

- 每行仍按各自行的查表结果填写
- 同时记录异常

## 9. 标题规则

### 9.1 标题内容

每一票标题格式固定为：

`装箱单（店铺简称-站点）海运第x票`

示例：

- `装箱单（WK-US）海运第1票`
- `装箱单（WK-US）海运第2票`

### 9.2 标题生成逻辑

标题中的 `店铺简称-站点` 来自：

1. `店铺简称`：取自 `店铺明细表.xlsx`
2. `站点`：从 `店铺+站点` 中提取最后一段站点代码

例如：

- `店铺简称 = WK`
- `店铺+站点 = WINKEAR-US`
- 标题标签 = `WK-US`

如果首行无法成功解析标题所需信息：

- 优先回退到当前票内首个可用的 `店铺简称 + 店铺+站点`
- 仍无法确定时使用 `待确认`
- 同时记录异常

## 10. 文件名规则

输出文件名格式为：

`店铺简称-品线-货代信息表（排序表）-yymmdd.xlsx`

示例：

`WK-落地浴缸-货代信息表（排序表）-260416.xlsx`

各段来源如下：

- `店铺简称`：取第 1 票查到的首个有效店铺简称
- `品线`：取第 1 票查到的首个有效原始品线值
- `yymmdd`：优先从第 1 票首行 `货件名称` 中提取 `YYYYMMDD` 后转成 `YYMMDD`

日期提取优先级：

1. `货件名称` 中的 `YYYYMMDD`
2. 源文件名中的 `YYYYMMDD`
3. 系统当前日期

如果目标文件名已存在，或保存时被占用，则自动回退为：

- `xxx_样式调整1.xlsx`
- `xxx_样式调整2.xlsx`

## 11. 样式规则

### 11.1 标题样式

每个票块的标题都要：

- 写入该票块的标题行第 1 列
- 合并 `A:M`
- 水平居中
- 垂直居中
- 字体：宋体
- 字号：20

### 11.2 表头样式

每个票块的表头都要：

- 使用两行表头结构
- 每列表头在上下两行做纵向合并
- 水平居中
- 垂直居中
- 自动换行
- 全边框

说明：

- 该规则适用于第 1 票以及后续所有票块
- 已修复第 2、3 票普通表头缺少上边框的问题

### 11.3 `箱号` 表头特殊规则

`箱号` 列表头显示为两行：

第一行：

- `箱号`

第二行：

- 当前票对应的 `FBA号`

第二行 `FBA号` 的视觉要求：

- 红色
- 加粗
- 宋体风格
- 小字号

### 11.4 数据区样式

所有新增写入的明细行和汇总行统一：

- 水平居中
- 垂直居中
- 四边框
- 字体：宋体
- 字号：12
- 行高：30 磅

### 11.5 汇总行规则

每票明细结束后追加 1 行汇总：

- 只填写 `套` 合计
- 只填写 `箱数` 合计
- 其他列留空
- 样式与新增数据区一致

## 12. 校验规则

处理完成后，至少校验以下内容：

1. 输出工作表数量必须为 1
2. 工作表名称必须等于总票数，例如 `3票`
3. 明细总行数必须等于所有来源有效行数总和
4. 每票 `套` 汇总必须等于来源 `发货量` 合计
5. 每票 `箱数` 汇总必须等于来源 `箱数` 合计
6. `箱号` 必须带 `编号` 前缀
7. `票数` 必须符合 `店铺简称海运第x票`
8. `说明书/包装上-品牌` 必须成功按查表规则填写，找不到时必须记异常
9. 每票标题必须符合 `装箱单（店铺简称-站点）海运第x票`
10. 每票标题必须保持 `A:M` 合并
11. 每票标题样式必须保持宋体 20 居中
12. 每票 `箱号` 表头必须显示对应 `FBA号`
13. 新增数据和汇总区域必须居中、带边框、宋体 12、行高 30

## 13. 异常处理规则

以下情况必须记录异常，不允许静默猜测：

- 找不到 `MSKU对应品线表.xlsx`
- 找不到 `店铺明细表.xlsx`
- 来源文件缺少必要字段
- `MSKU` 在映射表中找不到
- `店铺+站点` 在店铺明细表中找不到
- 同一个 `MSKU` 命中多条且无法自动唯一确认
- 同一票存在多个店铺简称
- 同一票存在多个 `店铺+站点`
- 同一票存在多个原始品线
- `箱号` 无法识别
- `FBA号` 无法判断
- 标题无法确定
- 日期只能通过回退规则取得
- 文件保存时需要回退名称
- 登录后仍停留在登录页
- FBA 搜索无结果
- FBA 搜索结果不唯一
- 货件详情页未识别到仓库卡片
- 未找到“下载装箱清单”按钮
- 未弹出“导出装箱清单”窗口
- 下载文件超时
- 一体化脚本启动浏览器失败

## 14. 当前运行入口

当前推荐生产入口是内网页面：

```text
http://127.0.0.1:8000/tasks/new
```

局域网同事访问时，将 `127.0.0.1` 替换为宿主机局域网 IP。

网页入口支持：

- 上传 `.txt` 清单
- 上传 `.xlsx` 清单
- 直接粘贴 FBA 号
- 后台排队执行
- 完成后下载业务结果文件

如果只有一个业务 `.xlsx` 结果，下载时直接返回单个 `.xlsx`；如果有多个业务 `.xlsx` 结果，下载时返回只包含业务结果文件的 zip。

### 14.1 Docker 常驻运行

```powershell
docker compose up -d --build
```

健康检查：

```powershell
curl http://127.0.0.1:8000/healthz
curl http://127.0.0.1:8000/api/system/status
```

### 14.2 命令行备用用法

#### 14.2.1 一体化执行方式

先安装依赖：

```powershell
python -m pip install -r .\requirements-rpa.txt
```

一体化常用执行方式：

```powershell
python .\lingxing_rpa_runner.py `
  --manifest E:\WoshiRPA\fba_manifest.txt `
  --resource-dir E:\WoshiRPA `
  --work-dir E:\WoshiRPA
```

可选参数：

- `config-file`：显式指定登录配置 JSON 文件；未传时默认读取 `resource-dir\lingxing_rpa.local.json`

运行结果目录结构固定为：

```text
<work-dir>
└─ runs
   └─ YYYYMMDD-HHMMSS
      ├─ batch_report.json
      ├─ <FBA号>
      │  ├─ automation_report.json
      │  ├─ downloads
      │  ├─ output
      │  └─ screenshots
      └─ ...
```

说明：

- 每个 FBA 独立下载到自己的 `downloads` 目录
- 每个 FBA 独立整理到自己的 `output` 目录
- 批量时某个 FBA 失败，不影响后续 FBA 继续执行
- 浏览器 profile 固定使用 `<work-dir>\chrome_profile_playwright`

#### 14.2.2 仅执行 Excel 整理

常用执行方式：

```powershell
python .\lingxing_excel_processor.py `
  --resource-dir E:\WoshiRPA `
```

- `output-dir`：输出结果目录

## 15. 当前业务口径总结

一句话总结当前规则：

支持通过 `网页任务/manifest -> Redis 排队 -> headless Playwright 自动化下载 -> 每个 FBA 独立整理` 的一体化流程生成业务结果文件；下载步骤严格按 `rpa.docx` 执行，整理步骤由代码直接生成输出工作簿，并继续遵循多票排版、查表映射、UPS 装箱信息分流、样式和校验规则。
