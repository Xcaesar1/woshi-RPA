# Harness 工作流说明

更新时间：2026-04-27

## 1. 总流程

```text
需求提交
  -> 资料检查
  -> 任务拆解
  -> 上下文注入
  -> 实现或修复
  -> 自动验证
  -> 人工验收
  -> 文档/测试/规则沉淀
  -> 提交代码
```

## 2. 需求提交

推荐输入：

- 原始文件或真实任务编号。
- 基础数据表。
- 期望结果文件。
- 当前错误截图或日志。
- 业务人员判断正确/错误的标准。

如果是浏览器自动化，还应提供：

- 目标平台地址。
- 测试账号交接方式。
- 关键页面截图。
- 典型业务编号。
- 可选 DOM 片段。

## 3. 资料检查

Agent 先确认：

- 输入文件是否能打开。
- 目标结果是否完整。
- 基础数据是否可用。
- 是否存在账号、隐私或不可逆操作风险。
- 是否能在当前环境复现。

资料不足但不影响局部推进时，先推进可完成部分。

## 4. 任务拆解

每次改动应拆成可验证的小目标：

- 页面或接口问题：先复现，再定位，再修复，再浏览器验证。
- Excel 问题：先找到源数据和目标差异，再修处理逻辑，再用样例回归。
- RPA 问题：先确认页面状态和 DOM，再修定位策略，再真实下载验证。
- 部署问题：先确认容器、日志、健康检查，再修配置。

## 5. 上下文注入

执行前优先读取：

- `docs/project/项目交付说明.md`
- `docs/business/领星自动化当前规则说明.md`
- `docs/harness/自动化需求交付规范.md`
- 本次任务相关代码和最近错误日志

不要只凭聊天记录修改。

## 6. 实现或修复

实现原则：

- 优先复用已有入口和模块。
- 不引入无必要复杂依赖。
- 不硬编码账号、路径、临时样例。
- 不绕过现有队列、目录隔离和浏览器单槽边界。
- 修改范围只覆盖本次需求。

## 7. 自动验证

按改动类型选择验证：

- 前端：构建、页面打开、表单提交、状态刷新、下载按钮。
- 后端：接口请求、任务创建、状态流转、异常返回。
- RPA：真实或受控样例下载、截图、报告。
- Excel：输出文件名、Sheet、字段、汇总、样式、异常报告。
- Docker：`web`、`worker`、`redis` 服务状态和健康检查。

无法验证时必须明确说明原因，不能把理论可行说成已验证。

## 8. 人工验收

人工主要验收：

- 结果文件能否直接使用。
- 业务含义是否正确。
- 页面是否好理解。
- 异常说明是否可排查。
- 是否符合当前工作习惯。

## 9. 资产沉淀

每次修复完成后至少判断是否需要沉淀：

- 新业务规则：更新 `docs/business/`。
- 新流程规则：更新 `docs/harness/`。
- 新失败类型：填写失败复盘。
- 可复跑样例：放入 `tests/fixtures/`。
- 可自动检查：加入 `scripts/` 或 `tests/regression/`。
