# 规则沉淀规范

更新时间：2026-04-27

## 1. 为什么要沉淀规则

RPA 项目的问题通常不是一次性 bug，而是“业务规则没有被系统记住”。如果修复只停留在代码里，后续换文件、换页面、换人员时仍然会重复踩坑。

因此每次稳定规则都要沉淀为长期资产。

## 2. 哪些内容必须沉淀

必须沉淀：

- 影响输出文件内容、样式、命名的规则。
- 影响页面自动化定位、下载、登录的规则。
- 影响任务状态、队列、并发、重试的规则。
- 影响安全、账号、隐私、权限的规则。
- 真实失败后确认过的修复经验。

可以不沉淀：

- 一次性的临时调试命令。
- 不会再次使用的本地路径。
- 没有业务含义的打印日志。
- 纯代码内部的小重构。

## 3. 沉淀到哪里

| 内容类型 | 位置 |
| --- | --- |
| 系统架构、部署、技术栈 | `docs/project/` |
| 领星业务规则、Excel 规则 | `docs/business/` |
| 需求提交、执行流程、复盘方法 | `docs/harness/` |
| 输入样例、目标样例 | `tests/fixtures/` |
| 回归测试 | `tests/regression/` |
| 检查器、批处理辅助脚本 | `scripts/` |

## 4. 文档规则

写规则时要做到：

- 写清楚适用范围。
- 写清楚判断条件。
- 写清楚异常情况。
- 写清楚示例。
- 避免只写“按之前规则”。

示例：

```text
错误写法：UPS 文件走第二种方式。

推荐写法：源文件内部“货件名称”包含 UPS，大小写不敏感，则走装箱信息输出；同一个 FBA 下载目录内的多个 UPS 文件合并成一份装箱信息工作簿，不同 FBA 之间不合并。
```

## 5. 测试规则

一条规则如果容易回归，应该尽量配一份样例或检查：

- 输入样例放 `tests/fixtures/`。
- 回归脚本放 `tests/regression/`。
- 结果检查器放 `scripts/`。

后续改动前，应优先跑相关回归，避免修复一个 FBA 又影响另一个 FBA。

## 6. PR 或提交前检查

每次提交前至少确认：

- 是否改了业务规则。
- 是否需要更新文档。
- 是否有样例可以加入回归。
- 是否有未验证内容。
- 是否误提交账号、密码、任务数据或大体积临时文件。
