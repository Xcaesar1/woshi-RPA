# 自动化与文档处理需求交付规范

更新时间：2026-04-27

本文档用于沉淀我们在领星 RPA 项目中形成的长期协作方式。后续不管是新的浏览器自动化、Excel/Word/PDF 处理、网页系统，还是客户端工具，都优先按这套方式交付。

核心原则是：**用户提供原始文件、基础数据和期望结果；开发智能体负责反推过程、实现逻辑、自动化验证和交付结果**。

---

## 1. 背景

过去如果按“第一步怎么点、第二步怎么点、第三步怎么整理”这种方式推进，会遇到几个问题：

- 信息传递慢，用户需要把大量细节拆成操作步骤。
- 页面、文件、业务规则经常有隐藏细节，口头描述容易遗漏。
- 智能体容易只执行局部步骤，而不是理解最终结果。
- 一旦换一个网站、换一种文件格式，又要重新解释大量过程。

领星 RPA 项目的实践证明，更高效的方式是：

```text
原始文件 / 原始页面
  + 基础数据表
  + 期望得到的结果文件
  + 少量业务约束
  -> 由智能体反推规则
  -> 实现自动化
  -> 对比结果
  -> 修正并固化规则
```

这能减少沟通损耗，也更适合后续长期迭代。

---

## 2. 目标

后续新需求默认达成以下效果：

- 用户不需要详细描述每一步操作。
- 用户只需要提供真实输入、基础数据和目标结果样例。
- 智能体负责分析文件结构、页面 DOM、业务字段、样式差异和命名规则。
- 实现结果要能复用、可测试、可继续扩展。
- 最终交付的不只是一次性脚本，而是可沉淀到系统里的能力。

---

## 3. 适用范围

适用于以下类型任务：

- 浏览器自动化，例如登录系统、搜索数据、点击按钮、下载文件、上传文件。
- Excel 自动处理，例如合并、拆分、查表、生成模板、设置样式、生成结果包。
- Word / PDF / 图片文档处理，例如提取内容、改格式、生成报告、批量转换。
- 内网页面任务化，例如上传文件、后台排队、状态查看、下载结果。
- 客户端或本地工具，例如一键处理文件夹、批量处理本地资料。
- 多系统串联，例如网页下载后再整理 Excel，再回传结果。

暂不适合作为唯一依据的任务：

- 涉及付款、公开发布、删除生产数据等不可逆操作。
- 涉及账号、密码、密钥、Cookie、客户隐私等敏感内容但没有安全方案。
- 目标结果本身不确定、业务人员也无法判断对错的需求。

---

## 4. 推荐输入包

以后提出新自动化或文档处理需求时，优先提供下面这些材料。

### 4.1 必须材料

```text
1. 原始文件
   实际会被系统处理的源文件，例如 xlsx、docx、pdf、txt、csv、图片等。

2. 基础数据
   查表用的数据、映射表、账号配置说明、字段解释、业务字典等。

3. 期望结果文件
   人工整理好的正确样例，最好来自同一批原始文件。

4. 结果判断规则
   哪些地方必须完全一致，哪些地方可以合理变化。
```

### 4.2 浏览器自动化额外材料

```text
1. 目标网址
2. 测试账号配置方式，不直接写进代码
3. 关键页面截图
4. 典型业务编号，例如 FBA 号、订单号、运单号
5. 需要点击或下载的最终结果说明
6. 如果可行，提供开发者工具中定位到的 DOM 片段
```

### 4.3 文件处理额外材料

```text
1. 源文件至少 2-3 个真实样例
2. 正确输出文件至少 1 个完整样例
3. 异常样例，例如缺字段、空数据、查不到映射、格式变化
4. 命名规则
5. 样式规则，例如字体、边框、颜色、合并单元格、行高、列宽
```

---

## 5. 智能体默认工作方式

拿到输入包后，智能体不应等用户逐步指挥，而应默认完成下面流程。

### 5.1 先理解，不盲改

- 读取项目结构和已有实现。
- 找到当前入口、核心处理函数、配置、测试方式和输出目录。
- 分析原始文件和目标文件之间的字段映射。
- 如果是网页自动化，先观察页面 DOM、URL、按钮、弹窗、下载行为。

### 5.2 反推业务规则

- 从目标结果反推字段来源。
- 从多个样例中找出稳定规则，而不是只适配单个文件。
- 区分“业务规则”和“偶然样式”。
- 对不确定规则做最小假设，并在交付时说明。

### 5.3 实现可复用能力

- 优先复用现有入口和模块。
- 不把临时样例路径、账号、密码硬编码进代码。
- 浏览器自动化优先使用稳定 DOM 定位，不依赖坐标点击。
- 文件处理优先按内容识别格式，不只依赖文件名。
- 输出逻辑要能处理批量、失败、部分成功和异常报告。

### 5.4 做真实验证

- 用用户提供的原始文件跑一遍。
- 生成结果后和期望结果对比。
- 对关键字段、文件名、样式、数量、汇总值做检查。
- 如果有 Docker / Web / Worker 链路，要验证容器内真实运行结果，不只验证本地脚本。

## 6. Playwright 浏览器自动化长期规则

后续浏览器自动化默认优先采用 Playwright，除非目标环境明确不支持。

### 6.1 定位原则

优先级如下：

```text
1. 可访问语义定位，例如 role、label、text
2. 稳定 DOM 属性，例如 name、placeholder、data-*、class 组合
3. 业务文本精确匹配，例如货件号、订单号、按钮文字
4. 结构关系定位，例如卡片内按钮、表格行内链接
5. 坐标点击作为最后兜底，不作为长期方案
```

### 6.2 执行原则

- 自动化前先确认登录态，失效后再走账号密码登录。
- 每个任务使用独立下载目录，避免串文件。
- 多人提交时，浏览器执行层默认单槽串行，避免同账号多浏览器并发风险。
- 下载后立即重命名，保留业务编号、顺序、仓库编码、文件类型等关键信息。
- 失败时保留当前 URL、页面标题、截图、错误原因和已下载文件清单。

### 6.3 页面变化处理

如果页面元素变化，不直接写死新坐标，应优先做：

- 扫描当前 DOM。
- 找到按钮、输入框、表格行、弹窗的稳定特征。
- 增加兼容选择器。
- 对关键动作加等待和校验。
- 保持页面打开或保存截图，方便排查。

## 7. 文档和表格处理长期规则

### 7.1 不从步骤出发，从结果出发

文件处理需求优先按下面方式理解：

```text
源文件里有什么
目标文件里要什么
中间需要查哪些基础表
哪些字段要计算
哪些样式必须保留
哪些异常要跳过或提示
```

不要让用户逐行解释“复制 A 列到 B 列”。

### 7.2 样式也是业务结果的一部分

如果目标是 Excel、Word 或 PDF，以下内容都要纳入验收：

- 文件名
- Sheet 名
- 标题
- 表头
- 合并单元格
- 字体
- 字号
- 字体颜色
- 背景色
- 边框
- 行高列宽
- 汇总行
- 打印或预览效果

### 7.3 异常要可解释

遇到以下情况不能静默失败：

- 查不到映射。
- 识别不到格式。
- 源文件缺列。
- 数据数量对不上。
- 输出为空。
- 样式或合并区域不符合预期。

系统应记录清楚：哪个文件、哪一行、哪个字段、为什么失败。

---

## 8. 推荐交付结构

复杂自动化需求建议最终沉淀为：

```text
业务入口
  Web 页面 / CLI / 客户端按钮

核心服务函数
  接收输入路径、基础数据目录、输出目录、配置路径

自动化模块
  负责网页操作、下载、上传、截图、失败恢复

处理模块
  负责解析源文件、查表、生成结果文件

报告模块
  负责记录成功失败、异常、产物路径

验证样例
  保留可复跑的输入样例和目标样例

文档
  固化输入格式、运行方式、业务规则和验收标准
```

---

## 9. 验收标准

一个新需求完成后，至少满足：

- 能用真实输入跑通。
- 输出文件能被业务人员直接使用。
- 关键字段和目标样例一致。
- 批量输入不会串文件、串目录、串结果。
- 失败时能继续处理其他任务。
- 页面或命令行有清楚的状态反馈。
- 配置、账号、密码、密钥不写死在代码里。
- 结果规则已经写入项目文档。

如果是网页自动化，还应额外满足：

- 登录态失效有处理方案。
- 页面定位不依赖单次截图坐标。
- 下载目录独立。
- 能保留失败截图和错误上下文。
- 多人同时提交时不会同时启动多个风险浏览器任务。

## 10. 后续需求提交模板

以后可以直接按这个模板给需求：

```text
需求名称：

我要实现的最终效果：

原始文件：
- 文件 1：
- 文件 2：

基础数据：
- 映射表：
- 账号/配置说明：

期望结果文件：
- 正确样例：

必须保持一致的地方：
- 字段：
- 文件名：
- 样式：
- 汇总：

可以由你自行判断的地方：
- 中间处理步骤
- 页面定位方式
- 代码结构
- 异常处理方式

验收方式：
- 用哪些文件跑
- 输出应该长什么样
- 哪些异常必须能处理
```

---

## 11. 关键经验总结

- 不要把用户变成流程录入员，用户最应该提供的是“输入”和“正确输出”。
- 自动化的价值不是模拟鼠标，而是把业务结果稳定产出。
- 浏览器自动化要读 DOM、等状态、验结果，不要只点坐标。
- 文档处理要从样例中反推规则，不要只按文件名猜格式。
- 任务系统要把多人提交和单槽执行分开，用户体验和生产稳定都更好。
- 长期可维护的系统必须有样例、报告、失败上下文和可复跑验证。
