# 领星 RPA Agent Harness 总方案

更新时间：2026-04-27

## 1. 背景

当前项目已经从“本地脚本”演进成“内网页面提交任务、后台队列执行、网页下载结果”的 RPA 系统。随着使用人数、业务规则和文件格式增加，如果每次出错都只临时修补，项目会越来越难维护。

Harness 的目标不是让 Agent 写完代码就结束，而是让它在一套可控流程里持续运行：拆解任务、注入上下文、调用工具、执行测试、接收审查反馈、修正错误、沉淀规则，并在关键节点让人类掌舵。

## 2. 核心目标

把 Agent 从“单次回答者”变成“在反馈中不断修正的执行者”。

具体目标：

- 每个需求都有标准输入：原始文件、基础数据、目标结果、验收标准。
- 每次执行都有可追踪产物：任务目录、日志、报告、结果文件。
- 每次失败都能复盘：原因、影响、修复、验证、沉淀。
- 每条稳定经验都要进入长期资产：文档、规则、测试样例、检查脚本或操作规范。
- 人类负责方向和验收，Agent 负责实现、验证和修正。

## 3. 当前 Harness 分层

```text
需求输入层
  钉钉宜搭 / 内网页面 / 原始文件 + 目标结果

上下文注入层
  docs/project / docs/business / docs/harness / 样例文件 / 历史失败案例

执行编排层
  FastAPI / Redis / Worker / Playwright / Excel Processor

校验反馈层
  结果文件检查 / 业务规则检查 / 日志报告 / 人工验收

长期资产层
  文档 / fixtures / regression tests / validators / PR checklist
```

## 4. 人类掌舵点

以下节点必须允许人类介入，不应该让 Agent 自行猜测：

- 业务目标不清楚，目标结果样例也无法判断对错。
- 涉及账号、密码、密钥、Cookie、付款、删除、公开发布。
- 新规则会改变现有用户的结果文件格式。
- 多个方案会明显影响后续维护成本或使用体验。
- 自动化站点出现验证码、风控或权限异常。

## 5. Agent 执行责任

Agent 默认负责：

- 阅读项目文档和已有代码。
- 识别现有入口、业务规则、测试方式和部署方式。
- 用真实输入或样例复现问题。
- 修改最小必要范围。
- 运行合适验证。
- 把修复经验沉淀到文档、测试或检查器。
- 最终给出完成结果、验证结果和剩余风险。

## 6. 第一阶段落地范围

第一阶段先做轻量 Harness，不引入复杂外部平台：

- 建立 `docs/harness/` 文档体系。
- 建立 `tests/fixtures/` 和 `tests/regression/` 目录。
- 建立 `scripts/` 目录，为后续检查器和批处理脚本预留位置。
- 后续每个稳定问题都至少更新一处长期资产。

暂不做：

- 自动创建 PR 和自动合并。
- 多 Agent 编排平台。
- 复杂权限系统。
- 自动替代人工业务验收。

## 7. 成功标准

后续如果再出现同类问题，应该能做到：

- 更快定位，因为文档有上下文。
- 更少误解，因为输入和目标结果标准化。
- 更少回归，因为有样例和检查器。
- 更容易交接，因为流程和规则都在仓库里。
