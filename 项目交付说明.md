更新时间：2026-04-24

## 1. 当前系统定位

本项目已经从最初的本地 Excel 处理脚本，扩展为一个可在局域网内使用的任务化 RPA 系统。

当前主流程是：

```text
同事打开内网页面
  -> 上传 FBA 清单 txt/xlsx
  -> 后端创建任务并写入 SQLite
  -> Redis 排队
  -> worker 串行获取浏览器执行槽
  -> 容器内 headless Playwright 登录领星并下载装箱清单
  -> 调用现有 Excel 整理逻辑
  -> 生成 zip 结果包
  -> 网页下载结果
```

这个形态适合当前阶段：多人可以同时提交任务，但真实浏览器自动化始终只允许一个任务执行，避免同一个领星账号被多个浏览器同时登录、搜索和下载。

## 2. 已实现功能

### 2.1 领星浏览器自动化

已实现真实领星站点的自动化下载链路：

- 自动打开领星 ERP 首页。
- 优先复用持久化登录态。
- 如果进入登录页，则读取 `lingxing_rpa.local.json` 自动输入账号密码登录。
- 进入 `FBA -> FBA货件`。
- 按清单中的 FBA 号搜索货件。
- 点击和 FBA 号完全一致的货件链接。
- 进入“箱子标签”页面。
- 识别每个仓库卡片。
- 点击下载装箱清单按钮。
- 选择“不包含图片”的导出方式。
- 等待下载完成并重命名。
- 每个 FBA 独立下载、独立整理、独立输出。

下载文件命名规则：

```text
FBA号_序号_仓库编码_NO_PIC.xlsx
```

示例：

```text
FBA19B9KWNFH_01_DFW6_NO_PIC.xlsx
FBA19B9KWNFH_02_DCA6_NO_PIC.xlsx
FBA19B9KWNFH_03_MDW6_NO_PIC.xlsx
```

### 2.2 Manifest 批量输入

支持两种输入文件：

- `.txt`：每行一个 FBA 号，忽略空行和 `#` 注释。
- `.xlsx`：读取第一个工作表，优先识别列名 `FBA号`、`货件单号`、`FBA`，找不到时回退第一列。

统一规则：

- 自动去空。
- 自动去重。
- 保持原始顺序。
- 某个 FBA 失败不会中断整个批次。

示例文件已保留在项目根目录：

- `fba_manifest.txt`
- `fba_manifest.xlsx`

- 部分成功任务：包含成功产物和失败说明。
- 失败任务：至少包含日志、错误说明、截图或报告。

### 2.5 Redis 队列与并发控制

当前采用：

- Redis：任务队列、浏览器执行槽锁、worker 心跳。
- SQLite：任务状态、路径、统计、错误信息等事实数据。
- worker：独立后台进程，从 Redis 取任务执行。

关键策略：

- 多人可以同时提交任务。
- 浏览器自动化全局只允许 1 个执行槽。
- 即使后续起多个 worker，也不会同时启动多个领星浏览器任务。
- worker 启动时会恢复异常状态：
  - `QUEUED` 但不在 Redis 队列里的任务会重新入队。
  - 心跳超时的 `RUNNING` 任务会标记为 `FAILED`。

### 7.2 Docker 环境变量

当前 `docker-compose.yml` 里主要配置：

```text
REDIS_URL=redis://redis:6379/0
PLAYWRIGHT_HEADLESS=1
BROWSER_MAX_CONCURRENCY=1
TASK_RETENTION_DAYS=30
WORKER_POLL_INTERVAL_SECONDS=5
HEARTBEAT_INTERVAL_SECONDS=15
SQLITE_JOURNAL_MODE=DELETE
```

说明：

- `BROWSER_MAX_CONCURRENCY=1` 是当前最重要的稳定性约束。
- `PLAYWRIGHT_HEADLESS=1` 表示容器内无界面运行浏览器。
- `SQLITE_JOURNAL_MODE=DELETE` 是为了避免 Windows 绑定目录下 SQLite WAL 不稳定。

职责：

职责：

职责：

- 获取浏览器执行槽。

页面使用 FastAPI 注入的 `page-data` 初始化，然后通过 API 轮询状态。

## 10. 当前边界与风险

### 10.1 只有一个领星账号

当前生产执行并发固定为 1。  
如果未来希望 2-3 个任务同时执行，需要先准备 2-3 个独立领星账号，再设计账号池和执行槽绑定。

不要直接把 `BROWSER_MAX_CONCURRENCY` 改成大于 1 来硬跑同一个账号。

### 10.2 领星页面 DOM 变化

自动化依赖领星页面的 DOM 和按钮结构。  
如果领星更新页面，可能需要重新定位：

- 登录输入框。
- FBA 菜单。
- 搜索框。
- FBA 链接。
- 下载按钮。
- 导出弹窗。
- 确定按钮。

### 10.3 验证码或额外安全校验

当前假设是：

- 账号密码有效。
- 登录不需要验证码。
- 容器 headless 登录不会被风控拦截。

如果未来出现验证码或二次验证，需要增加人工介入方案或改成可视化调试方案。

### 10.4 Docker Desktop 依赖

`restart: unless-stopped` 只在 Docker Desktop 正常运行时生效。  
如果 Windows 重启后 Docker Desktop 没有自动启动，服务也不会自动恢复。

建议在宿主机上开启 Docker Desktop 开机自启。

### 10.5 数据保留

当前设计保留 30 天任务结果。  
如果任务量变大，需要定期关注：

- `data/jobs`
- `data/results`
- Docker volume `redis-data`

## 12. 维护提醒

这个项目最重要的稳定性原则是：

```text
网页提交可以并发，真实浏览器执行必须受控。
```

现在的实现已经把“用户体验”和“站点风险”分开了：用户侧不会因为别人正在跑任务而无法提交；执行侧也不会因为多人同时点提交而同时打开多个浏览器冲击同一个领星账号。

后续扩展时，优先保护这个边界。
